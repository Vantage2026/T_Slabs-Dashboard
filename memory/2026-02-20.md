### Dependency Sync Implementation (from emmy/DEPENDENCY_SYNC_REQUIREMENTS.md)
I wrote the original spec, Emmy reviewed it, and I'm implementing her requirements.

**Completed:**
1. **Prisma schema** — `TaskDependency` model added with `pendingTargetExternalId` for two-pass resolution. Canonical direction only (no inverse storage, per Emmy). Relations on Task model.
2. **Types** — `NormalizedDependency` interface added to `NormalizedTask` in `src/types/index.ts`
3. **Jira adapter** — `normalizeIssue` now builds `dependencies: NormalizedDependency[]` from issue links, including `externalLinkId` from raw link objects for stale cleanup
4. **Sync engine** (`src/lib/sync/engine.ts`):
   - `syncDependencies()` — creates/updates TaskDependency rows per task, handles pending targets with find-or-create (avoids NULL unique constraint issue)
   - `resolvePendingDependencies()` — second pass after full project sync resolves pending links
   - `cleanStaleDependencies()` — removes TaskDependency rows whose externalLinkId is no longer present in source
   - Wired into `upsertTask` (calls syncDependencies) and `syncAllTasksInbound` (calls resolve + cleanup after sync)
5. **Critical path engine** (`src/lib/critical-path.ts`):
   - Added `ResolvedDependency` interface
   - `computeCriticalPath()` now accepts optional `resolvedDeps` param
   - Uses resolved deps for blocking counts and chain length when available, falls back to metadata
   - New `computeBlockerChainFromDeps()` uses internal task IDs instead of Jira keys
6. **Horizontal cascade** (`src/lib/timeline/propagate-cascade.ts`):
   - New `propagateDependencyCascade()` — when blocker due date slips past blocked task's start, records a cascade event
   - `recordDateChangeAndPropagate()` now runs both vertical (parent) and horizontal (dependency) cascade
